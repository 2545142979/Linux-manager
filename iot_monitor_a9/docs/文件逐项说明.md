# IoT Monitor A9 文件逐项说明

本文档按当前工程目录，对主要文件逐个说明它的职责、关键内容以及它和其它模块的关系。

## 1. 工程整体作用

这个项目是运行在 A9 Linux 板子上的服务端程序，负责做三件事：

1. 从摄像头持续采集 JPEG 图像。
2. 从串口持续接收 M0 上报的环境数据包。
3. 通过 TCP 向 Qt 客户端提供图片、环境数据，并转发控制命令。

可以把它理解成“摄像头 + 串口 + 网络”的中间层。

## 3. 程序启动流程

程序主流程如下：

1. `main.c` 初始化共享状态和串口上下文。
2. `main.c` 启动串口线程 `serial_thread()`。
3. `main.c` 启动摄像头线程 `camera_thread()`。
4. `main.c` 创建 TCP 监听套接字。
5. 每来一个客户端连接，就创建一个 `tcp_client_thread()` 单独处理。
6. TCP 线程根据客户端请求，去共享区取图片或环境数据，或者把命令下发到串口。

## 4. 协议说明

本项目里的“协议”指的是 A9、M0、Qt 之间约定好的数据包格式。

### 4.1 为什么有 `16`、`32`、`64`

协议文档里出现的 `uint16`、`uint32`、`uint64`，表示的是字段的数据宽度，不是协议版本号。

- `uint16`
  无符号 16 位整数，占 2 字节。
- `uint32`
  无符号 32 位整数，占 4 字节。
- `uint64`
  无符号 64 位整数，占 8 字节。

之所以要区分这些类型，是因为不同字段需要的取值范围不同。

例如：

- 温度、湿度通常范围不大，用 `uint16` 就够了。
- 光照、状态、加速度通常需要 4 字节，所以用 `uint32`。
- `adc` 原始值可能比较大，协议直接给了 8 字节，所以用 `uint64`。

换句话说，协议不是随便写成 `16` 或 `64`，而是在定义“这个字段应该占多少个字节”。

### 4.2 为什么 `uint16` 是 2 字节，`uint64` 是 8 字节

1 个字节等于 8 位，所以：

- 16 位 = 2 字节
- 32 位 = 4 字节
- 64 位 = 8 字节

因此在代码里会看到：

- `protocol_read_le16()`
  从 2 个字节中读一个整数。
- `protocol_read_le32()`
  从 4 个字节中读一个整数。
- `protocol_read_le64()`
  从 8 个字节中读一个整数。

比如环境包中的：

- `buf[4]` 和 `buf[5]` 一起组成温度 `tmp`
- `buf[6]` 和 `buf[7]` 一起组成湿度 `hum`
- `buf[12]` 到 `buf[19]` 一共 8 个字节组成 `adc`

### 4.3 小端序是什么意思

协议文档里说明长度字段和各数值字段采用小端格式。

小端序的意思是：低字节放在前面，高字节放在后面。

例如十六进制数值 `0x1234`：

- 小端存储是 `34 12`
- 大端存储是 `12 34`

所以协议里长度 `36` 写成 16 位小端时，就是：

- 十六进制 `0x0024`
- 字节流写成 `0x24 0x00`

这就是为什么文档里会写“`0x24 00` 代表 36 字节”。

### 4.4 36 字节环境包布局

环境包固定 36 字节，格式如下：

```text
Byte 0      : Tag        1字节   0xBB
Byte 1      : ID         1字节   0x0D
Byte 2-3    : Len        2字节   固定 36，小端
Byte 4-5    : tmp        2字节   uint16
Byte 6-7    : hum        2字节   uint16
Byte 8-11   : acc        4字节   uint32
Byte 12-19  : adc        8字节   uint64
Byte 20-23  : light      4字节   uint32
Byte 24-27  : state      4字节   uint32
Byte 28-31  : padding    4字节   预留
Byte 32-35  : crc        4字节   uint32
```

这里最容易混淆的是 `adc`，因为它一个字段就占了 8 个字节，所以必须用 `uint64` 来接。

### 4.5 36 字节命令包布局

命令包也是固定 36 字节，结构和环境包前 4 个字节一致：

```text
Byte 0      : Tag        1字节   0xDD
Byte 1      : ID         1字节   0x0D
Byte 2-3    : Len        2字节   固定 36，小端
Byte 4      : opcode     1字节   控制码
Byte 5-35   : 其余保留   31字节  当前填 0
```

当前项目里主要使用 `Byte 4` 来表示控制动作：

- `0x00` LED_ON
- `0x01` LED_OFF
- `0x02` BUZ_ON
- `0x03` BUZ_OFF
- `0x04` FAN_ON
- `0x08` FAN_OFF

### 4.6 代码里如何解析这些字段

协议解析主要在 `src/protocol.c` 中完成。

几个关键函数的含义是：

- `protocol_read_le16()`
  从 2 字节小端数据读出一个 `uint16`
- `protocol_read_le32()`
  从 4 字节小端数据读出一个 `uint32`
- `protocol_read_le64()`
  从 8 字节小端数据读出一个 `uint64`
- `protocol_parse_env_packet()`
  按协议定义把 36 字节环境包拆成结构体字段
- `protocol_build_control_packet()`
  按协议定义把控制命令拼成 36 字节命令包

例如 `protocol_parse_env_packet()` 做的事情，本质上就是：

```text
tmp   = 读 buf[4..5]
hum   = 读 buf[6..7]
acc   = 读 buf[8..11]
adc   = 读 buf[12..19]
light = 读 buf[20..23]
state = 读 buf[24..27]
crc   = 读 buf[32..35]
```

### 4.7 一个简单例子

假设环境包前几字节是：

```text
BB 0D 24 00 19 00 50 00
```

那么可以这样理解：

- `BB`
  说明这是环境包。
- `0D`
  设备 ID 是 `0x0D`。
- `24 00`
  小端整数，表示总长度 36。
- `19 00`
  小端整数，温度值是 25。
- `50 00`
  小端整数，湿度值是 80。

这就是协议里“16 位字段”的实际表现方式。

## 5. 逐个文件说明

### 5.1 `CMakeLists.txt`

这是整个项目的 CMake 构建入口。

它主要做了几件事：

- 指定项目名称为 `iot_monitor_a9`。
- 指定语言为 C，标准为 `C11`。
- 把 `src/` 下的 6 个 `.c` 文件加入编译。
- 生成目标程序 `iot_monitor_server`。
- 把 `include/` 加入头文件搜索路径。
- 对 `GCC/Clang` 开启 `-Wall -Wextra -Werror`。
- 链接 `pthread`，因为项目使用了多线程。

可以把它看成“告诉 CMake 这个项目由哪些源码组成，最后产出什么可执行文件”。

### 5.2 `README.md`

这是项目的简要使用说明。

它主要面向“第一次接手项目的人”，介绍：

- 项目是做什么的。
- 工程包含哪些核心模块。
- 怎么本地编译。
- 怎么 ARM 交叉编译。
- 程序如何运行。
- TCP 侧支持哪些请求。

README 适合快速上手，本文档则更适合读源码时查每个文件的职责。

### 5.3 `cmake/toolchains/arm-linux-gcc.cmake`

这是 ARM 交叉编译用的 CMake 工具链文件。

它的作用是告诉 CMake：

- 目标系统是 `Linux`。
- 目标处理器是 `arm`。
- 编译器默认使用 `arm-linux-gnueabi-gcc`。

如果没有这个文件，CMake 会默认用本机编译器，而不会切换到 ARM 交叉工具链。

### 5.4 `scripts/build_linux.sh`

这是本机 Linux 编译脚本。

它做的事情很直接：

- 计算工程根目录。
- 把构建目录固定到 `build/linux`。
- 默认按 `Release` 模式配置。
- 调用 `cmake -S ... -B ...` 生成构建文件。
- 调用 `cmake --build ...` 执行编译。

这个脚本的意义是把常用命令固定下来，避免每次手动敲长串 CMake 参数。

### 5.5 `scripts/build_arm_linux_gcc.sh`

这是 ARM 交叉编译脚本。

它和本机脚本类似，但多了两个动作：

- 先检查 `arm-linux-gnueabi-gcc` 是否存在。
- 把 `cmake/toolchains/arm-linux-gcc.cmake` 作为工具链文件传给 CMake。

如果你的交叉编译器不在 `PATH` 里，可以通过环境变量 `ARM_LINUX_GCC_COMPILER` 指定。

### 5.6 `include/protocol.h`

这是协议层头文件，用来统一管理 36 字节通信协议的常量、结构体和函数声明。

它定义了：

- 包长 `PROTOCOL_PACKET_SIZE = 36`
- 设备 ID `PROTOCOL_DEFAULT_DEVICE_ID = 0x0D`
- 环境包、命令包、刷卡包的 Tag
- 控制命令枚举 `enum control_opcode`
- 环境数据结构 `struct env_data`

`struct env_data` 里既有解析后的字段，也保存了一份 `raw[36]` 原始字节流。这样 TCP 收到 `env` 请求时，可以直接把原始 36 字节发回去，不需要重新拼包。

### 5.7 `include/shared.h`

这是共享状态头文件。

它定义了两个核心共享对象：

- `struct image_frame`
  保存最新一帧 JPEG 图像。
- `struct shared_state`
  保存全局共享状态，里面有图片、环境数据以及对应的互斥锁。

这个模块的设计目的，是让摄像头线程、串口线程和 TCP 线程之间不要互相直接操作底层设备，而是统一通过共享区交换“最新数据”。

### 5.8 `include/serial_port.h`

这是串口模块头文件。

它定义了：

- `struct serial_context`
  保存串口文件描述符、互斥锁、设备路径、波特率。
- `struct serial_thread_args`
  串口线程启动时需要的参数。

它还声明了 4 类接口：

- 串口上下文初始化和销毁。
- 原始字节发送。
- 文本命令转串口命令包。
- 串口线程入口函数。

### 5.9 `include/camera.h`

这是摄像头模块头文件。

它定义了 `struct camera_thread_args`，里面包括：

- 共享区指针。
- 摄像头设备路径。
- 分辨率。
- 预热丢帧数量。
- 采集间隔。

这个头文件比较薄，主要是为 `camera_thread()` 提供参数声明。

### 5.10 `include/tcp_server.h`

这是 TCP 模块头文件。

它定义了：

- `struct tcp_client_args`
  每个客户端线程需要的参数，里面有连接套接字、共享状态和串口上下文。

它还声明了两个关键接口：

- `tcp_server_listen()`
  创建监听套接字。
- `tcp_client_thread()`
  处理单个客户端连接。

### 5.11 `src/main.c`

这是程序入口文件，也是线程和模块的总装配点。

它主要负责：

- 解析命令行参数。
- 初始化 `shared_state`。
- 初始化 `serial_context`。
- 启动串口线程。
- 启动摄像头线程。
- 创建 TCP 监听端口。
- 每接受一个客户端连接，就启动一个客户端处理线程。

这个文件不做具体业务解析，它的重点是“把各模块连起来”。

其中 `parse_port()` 的作用是把命令行字符串安全地转成端口号，非法输入会回退到默认值 `9527`。

### 5.12 `src/protocol.c`

这是协议层实现文件。

它负责做两类事情：

- 字节序处理。
- 命令包/环境包的编解码。

关键函数说明：

- `protocol_read_le16()` / `protocol_read_le32()` / `protocol_read_le64()`
  从小端字节流读取整数。
- `protocol_write_le16()`
  把 16 位整数写成小端格式。
- `protocol_is_command_packet()`
  判断一段 36 字节数据是否是合法命令包。
- `protocol_parse_env_packet()`
  把环境包解析成 `struct env_data`。
- `protocol_build_control_packet()`
  根据操作码构造 36 字节命令包。
- `protocol_opcode_from_text()`
  把 `led_on`、`fan_off` 这种文本命令映射成协议里的操作码。

这个文件是整个项目里“最不依赖硬件”的模块，主要负责协议规则本身。

### 5.13 `src/shared.c`

这是共享状态实现文件。

它的作用是给多线程访问提供一个统一、安全的数据缓存层。

关键函数说明：

- `shared_state_init()`
  初始化共享状态和互斥锁。
- `shared_state_destroy()`
  销毁互斥锁。
- `shared_state_store_image()`
  摄像头线程写入最新 JPEG。
- `shared_state_copy_image()`
  TCP 线程读取最新 JPEG。
- `shared_state_store_env()`
  串口线程写入最新环境数据。
- `shared_state_copy_env()`
  TCP 线程读取最新环境数据。

这个文件的核心价值不是“逻辑复杂”，而是“把线程同步封装起来”。这样其它模块就不用自己重复写锁逻辑。

### 5.14 `src/serial_port.c`

这是串口模块实现文件，负责和 M0 通信。

可以把它分成 4 个部分理解：

#### 1. 串口配置

- `serial_configure_fd()`

设置波特率、数据位、停止位、校验位、原始模式等串口参数。当前配置是：

- 波特率 `115200`
- 8 数据位
- 1 停止位
- 无校验
- 无硬件流控

#### 2. 串口连接管理

- `serial_open_locked()`
- `serial_close_locked()`
- `serial_context_init()`
- `serial_context_destroy()`

这一部分负责打开串口、保存文件描述符、初始化互斥锁，以及在异常时关闭并重连。

#### 3. 串口发送

- `serial_send_bytes()`
  发送原始字节流。
- `serial_send_command_text()`
  把文本命令先转成协议包，再通过串口发送。

#### 4. 串口接收线程

- `serial_thread()`

这是一个长期运行的线程，逻辑是：

1. 如果串口还没打开，就尝试打开。
2. 先读 1 个字节作为包头。
3. 只有当包头是 `0xBB` 时，才继续读剩下 35 字节。
4. 调用 `protocol_parse_env_packet()` 解析环境包。
5. 把解析结果写进共享区。
6. 如果串口异常，就关闭并延时重试。

这个线程对应需求里的“环境数据线程”。

### 5.15 `src/camera.c`

这是摄像头模块实现文件，负责通过 V4L2 持续采图。

它主要包含下面几部分：

#### 1. 设备与格式准备

- `camera_open_device()`
- `camera_require_mjpeg()`

这里会打开 `/dev/video0`，检查设备是否支持视频采集和流模式，并强制要求输出 `MJPEG`。这样后面拿到的数据可以直接作为 JPEG 发送给 Qt。

#### 2. 缓冲区准备

- `camera_prepare_buffers()`

这里通过 `VIDIOC_REQBUFS`、`VIDIOC_QUERYBUF`、`mmap`、`VIDIOC_QBUF` 建立 V4L2 的内核缓冲区和用户态映射。

#### 3. 采集控制

- `camera_start_stream()`
- `camera_wait_frame()`
- `camera_capture_one()`
- `camera_requeue()`
- `camera_cleanup()`

这一组函数负责启动视频流、等待帧到来、出队一帧、重新入队以及释放资源。

#### 4. 线程主循环

- `camera_thread()`

这是摄像头线程入口，逻辑是：

1. 打开并初始化摄像头。
2. 启动视频流。
3. 丢弃前几帧启动脏数据。
4. 进入循环，不断获取一帧 JPEG。
5. 把最新 JPEG 写进共享区。
6. 重新入队缓冲区。
7. 按设定间隔稍作休眠。

这个线程对应需求里的“视频采集线程”。

### 5.16 `src/tcp_server.c`

这是 TCP 业务模块实现文件，负责和 Qt 客户端交互。

可以分 3 层理解：

#### 1. 底层发送工具

- `write_all()`

它保证缓冲区内容被完整写出，避免 `write()` 一次只写部分数据。

#### 2. 各类请求处理函数

- `handle_picture_request()`
  从共享区取最新 JPEG，先发 7 位 ASCII 长度，再发图片内容。
- `handle_env_request()`
  从共享区取最近一次环境包，并原样返回 36 字节。
- `handle_text_command()`
  处理文本命令，例如 `led_on`、`fan_off`。
- `trim_text_command()`
  去掉文本命令结尾的空白字符。

#### 3. 服务端主逻辑

- `tcp_server_listen()`
  创建 TCP 监听套接字。
- `tcp_client_thread()`
  处理单个客户端连接。

`tcp_client_thread()` 的处理顺序是：

1. 如果收到的是 `pic`，走图片流程。
2. 如果收到的是 `env`，走环境包流程。
3. 如果收到的是合法 36 字节命令包，直接转发给串口。
4. 否则按文本命令处理。

这个线程对应需求里的“TCP 业务线程”。

## 7. 文件之间的依赖关系

可以用下面这条主链来理解整个工程：

```text
main.c
  ├── shared.c / shared.h
  ├── serial_port.c / serial_port.h
  ├── camera.c / camera.h
  └── tcp_server.c / tcp_server.h

serial_port.c
  └── protocol.c / protocol.h

tcp_server.c
  ├── shared.c / shared.h
  ├── serial_port.c / serial_port.h
  └── protocol.c / protocol.h

shared.c
  └── protocol.h
```

依赖关系的意思是：

- `main.c` 负责把所有模块组织起来。
- `protocol.c` 负责协议细节，供串口和 TCP 共用。
- `shared.c` 提供线程之间交换数据的公共区域。
- `serial_port.c` 和 `camera.c` 负责采集最新数据。
- `tcp_server.c` 负责把这些数据提供给客户端。

2. 再看 `src/main.c`，掌握主流程和线程模型。
3. 看 `include/protocol.h` 和 `src/protocol.c`，理解通信协议。
4. 看 `include/shared.h` 和 `src/shared.c`，理解线程共享数据方式。
5. 看 `src/tcp_server.c`，理解客户端怎么交互。
6. 看 `src/serial_port.c`，理解 M0 数据怎么接入。
7. 最后看 `src/camera.c`，理解 V4L2 采图细节。

这样读下来，理解成本最低。
